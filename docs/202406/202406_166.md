今天我们聊聊，日本的 IT 行业。

日本 IT 行业正在尝试，摆脱原有的分包业务结构，尝试自主研发并为此拥抱敏捷（Scrum）开发方法。

> # 个人观察——日本 IT 行业内部分类

在日本，IT 行业中把公司大致分为两类：

*   提供 IT 咨询和解决方案的公司 *   自主开发服务的公司 

提供 IT 咨询和解决方案的知名企业有 NTT DATA 和野村综合研究所等。

自主开发的企业如乐天株式会社和 Yahoo（雅虎）株式会社，多为创业公司或大型创业公司。

这些提供 IT 咨询和解决方案的企业，被认为是 IT 开发生态链中的“上游工程”企业。业内通常称为 SIer。

此外，还有一些公司负责“下游工程”（写代码，实现功能），其中大部分是中小企业或专门从事这类工作的公司。

在我观察的范围内，“上游工程”企业相对不受欢迎。

有人认为提供 IT 咨询和解决方案的上游无法学到技术，下游则是黑暗工作环境”。

为什么会有这种说法呢？

这与日本特有的 IT 企业问题有关。

> # 日本与美国的差异

日本的 SIer 常被拿来与美国企业内的 IT 工程师进行比较。

美国的大多数 IT 工程师在公司内部的 IT 部门进行工作。

相反，**日本的 IT 工程师主要在以 IT 为主业的公司工作。**

原因是日本企业如果雇用 IT 工程师后，由于种种原因决定不再对 IT 项目进行投资时，由于劳动法的限制，他们无法轻易解雇员工。

因此，日本的传统企业会成立系统专门的子公司，在那里雇用 IT 工程师。

这些子公司被称为"用户系 SIer"，例如，以下公司就是"用户系 SIer"：

*   伊藤忠科技解决方案公司（伊藤忠商事株式会社） *   日铁解决方案公司（日本制铁株式会社） *   SCSK 公司（住友商事株式会社） 

由于，这些"用户系 SIer"公司是由，非 IT 为主业的传统公司投资设立的子公司，规模也往往是相对较大的。

相对来说，日本的 IT 工程师大部分都在进行 SIer 工作。

> # 日本独有的 IT 业务分包结构

日本的 SIer 有一个特点是分包结构。

为什么会形成分包结构呢？

其中一个原因是大 SIer 的销售能力强，无论内部资源如何都能拿到合同。

如果内部资源不足，就会转给分包商（称为合作伙伴），从而能够进行系统开发。

需求定义、设计、人员配置、管理等工作由大 SIer 负责，结果是实施和测试等所谓的下游工程交给分包企业。

分包企业是中型 SIer 或中小开发公司、系统工程服务（SES）公司。

如今有很多编程学校和编程学习服务，想成为程序员的人很多，因此其覆盖面广，即使原承包企业没有资源也能完成项目。

此外，这种分包结构适合按顺序进行需求定义、基本设计、详细设计、实施、单体测试、集成测试的瀑布开发方法。

## 分包结构的问题点

**1.导致质量下降或延迟交付**

分包结构自然有很多问题，这也是一些 IT 工程师不喜欢 SIer 的原因。

开发实际由分包企业的 IT 工程师完成，作为用户企业，很难掌握末端 IT 工程师的技术水平。

对用户企业来说，窗口是负责的大 SIer，所以如果有进度延迟，他们会向 SIer 抱怨。

而 SIer 则会向分包企业施压。

分包企业常用大量新手，因此写出高质量代码的可能性不大，尤其是在时间紧迫的情况下。

**2.无法积累内部知识**

与自主开发不同，实际开发不是由自己员工进行，无法在公司内部积累知识。

**3.内部缺乏懂系统的人会被剥削**

用户企业如果引入 ERP 等企业运营系统，换系统很困难。

仅修改一个按钮就可能收取数百万至数千万日元这种事情也很常见。

**4.开发出对终端用户使用极其不方便的系统**

分包结构下，直接客户是上游公司，考虑业务时，提供对直接客户正确的东西比对终端用户方便更重要。

因此，如今在网上银行系统中仍能看到“错误代码：001”这样的提示。

对终端用户来说，这是无关紧要的信息，但在设计时为了减少公司间的误解，可能采用了这种规格。

> # IT 行业需要改变

正因为这些问题，日本的 IT 被认为落后，很多日本本土 IT 工程师对 SIer 持反感态度。

日本企业，尤其是制造业，仍然具有很大的潜力。

制造业等企业目前组件重视内部培训和拥有 IT 团队，自主开发和研发系统。

由于之前落后的比较多，日本的主流仍将是与 SIer 合作。

然而，摆脱分包结构仍是必要的。

以 AI 企业为例，小公司独立开发大型企业系统是可能的。

AI 企业能够做到这一点的一个原因是采用了敏捷（Scrum）开发方法。

敏捷开发是逐步构建可发布系统的方法，与瀑布开发形成对比。

**SIer 要变革，目前业内正在采用敏捷开发方法。**

但这需要克服一些问题。

第一，传统企业需要对开发有一定了解。敏捷开发在短时间内反复进行开发和确认，用户企业无法完全委托。此外，系统开发完成后，企业内部需积累运营知识，否则与之前的开发无异。因此，这将增加一定负担。

第二，IT 工程师的人才短缺。敏捷开发要求小团队成员能在多个领域进行开发，每个人的要求水平更高。仅通过几个月编程学校学习的 IT 工程师可能无法胜任。因此，人材短缺是一个问题。相对来说，IT 工程师的地位可能上升。
